# F-format-strategy — стратегия общения с локальными моделями

Сквозная фича. Описывает соглашения, по которым приложение запрашивает у локальных Ollama-моделей текстовые артефакты, и парсит их ответы.

## Контекст

Раньше мы пробовали JSON-schema через параметр `format` в Ollama API. Опытная проверка:

- Mistral 3.2 и Gemma 4 — нормально подчиняются переданной схеме.
- Qwen 3.6 — игнорирует схему: возвращает массив верхнего уровня, добавляет markdown-блоки `\`\`\`json`, использует произвольные имена полей.

Поскольку модели нужны все (Mistral отлично работает на простых случаях, но как он справится с гигантскими промптами на этапе игрового цикла — неизвестно), нужен формат, который одинаково хорошо понимают **все** локальные модели.

## Решение

Отказываемся от JSON-schema. Все ответы — в едином текстовом формате «один артефакт = один блок в квадратных скобках с парами KEY=value».

### 1. Формат записи

```
[
NAME=Короткое название
TAGLINE=Одно предложение для обложки.
]
```

Правила:

- Запись окружена `[` и `]`, каждая скобка — на своей строке.
- Каждое поле — это `KEY=value`, всё что после первого `=` до `\n` — значение.
- Ключи — латиница верхнего регистра, без пробелов: `NAME`, `TAGLINE`, `DESCR`.
- Если значение многострочное, используется heredoc:

```
[
NAME=Хроники Зелёного Руина
DESCR<<<
Когда города поглотил лес,
а древние технологии стали ядом —
последний ключ от Земли спасёт оставшихся.
>>>
]
```

  - Открывается `KEY<<<` (на той же строке что и ключ), закрывается `>>>` на отдельной строке.
  - Внутри допустимы любые символы, кроме одинокой строки `>>>`.

- Парсер **толерантный**: игнорирует всё, что вне `[…]`, пустые строки и пробелы в начале/конце, BOM, markdown-обёртки `\`\`\``.

### 2. Один запрос = один артефакт

Главный принцип:

> Приложение знает, какие артефакты уже созданы. Оно передаёт их в контексте промпта и просит у модели **ровно один** новый артефакт.

Иерархия (например, дерево локаций) собирается **в приложении**, а не моделью. Для каждой новой ноды дерева — отдельный вызов модели. Контекст вызова содержит уже существующие соседние и родительские ноды.

**Исключение** — этап выбора варианта (`WorldHeader`). Здесь нужно 3 альтернативы, чтобы игрок выбрал одну. Это один вызов с 3 записями подряд. Везде в других местах — один артефакт за вызов.

### 3. Идентификаторы артефактов делает приложение

`ArtifactId` (например, `World.KhronikiZelenogoRuina`) и slug формирует C#-сервис, а не модель. Источник slug — русское `NAME`, прогнанное через транслитерацию (см. `Services/Slug/Slugger.cs`).

Раньше модель сама выдумывала slug в JSON-поле `slug`. Это убираем — модель ошибалась со схемой, а slug всё равно дублирует `NAME`.

### 4. Структура парсера

Универсальный парсер живёт в `Services/KeyValue/KvParser.cs`:

```csharp
KvParser.ParseSingle(rawText) -> KvRecord
KvParser.ParseAll(rawText)    -> IReadOnlyList<KvRecord>
KvRecord["NAME"]              -> string
KvRecord.TryGet("DESCR", out var v)
```

`KvRecord` — словарь `IReadOnlyDictionary<string, string>` с case-insensitive ключами.

Промпт-помощники живут рядом в `Services/KeyValue/KvFormatter.cs`:

```csharp
KvFormatter.BuildFormatInstruction(fields)
KvFormatter.BuildExample(values)
```

Они генерируют единообразный кусок промпта с описанием формата и примером, чтобы не дублировать его в каждой фиче.

## Что меняется в существующем коде

- `WorldHeader`: убираем `JsonSchema()`, заменяем системный промпт на kv-инструкции, убираем терпимый JSON-парсер, используем `KvParser.ParseAll` (ждём 3 записи).
- `WorldOverview`: убираем поле `slug` из ответа модели (slug строит приложение), убираем `JsonSchema()`, парсер — `KvParser.ParseSingle`. Поле `description` многострочное → heredoc.
- `OllamaClient.GenerateAsync` параметр `formatSchema` оставляем — он опциональный, не мешает; со временем уберём.

## Что меняется в будущих фичах

- F-1.3 (уровни локаций): для каждой локации — отдельный вызов с контекстом «уже созданные локации в этом мире». Модель возвращает одну запись `[NAME=... DESCR=...]`. Приложение строит `ArtifactId` вида `WorldPartL1.{WorldSlug}.{LocationSlug}`.
- F-1.4+ (Население, Предыстория и т.д.) — то же.

## Открытые вопросы / на потом

- Если значение должно содержать строку `>>>` — нужно экранирование. Пока считаем, что это краевой случай и проигнорируем.
- Если модель пропустит обязательное поле — повторный вызов или ошибка пользователю? Сейчас в `WorldOverviewService` есть retry-цикл (3 попытки), оставляем эту схему. В `WorldHeaderService` retry не было — добавим аналогичный.
